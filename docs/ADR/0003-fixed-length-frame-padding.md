# ADR-0003: 256 バイト固定長フレーム v1 の採用

- **ステータス**: 採択 (Accepted)
- **日付**: 2026-05-06 (v0.1 初版), 2026-05-06 (v0.2 で v1 構造に拡張), 2026-05-06 (v0.2 audit-sync)
- **関連**: ADR-0001 (スコープ), ADR-0004 (SAS), 設計指針 §6

## コンテキスト

QUIC は内容を暗号化するが、暗号化されたペイロード自体のバイト数は観測可能
である。これにより、観測者は「6 バイトのメッセージは Yes/No 的な短い応答」
「200 バイトのメッセージはある程度の長文」といった推論が可能になる。

トラフィック分析対策として、メッセージを固定長に揃える「パディング」が
古くから研究されている。固定長は実装が単純な反面、最大メッセージ長を超える
とフレーム分割が必要になり、また短いメッセージに対しては帯域効率が悪い。

v0.1 では `[len: 2B][payload][padding]` の単純な構造を採用したが、
SAS ハンドシェイク・将来のダミーフレーム・制御信号を区別するためには
フレーム種別の情報が必要である。

なお、設計指針 §6.2 が明記するとおり、本 ADR の保護範囲は
**アプリケーション層の各メッセージを 256 バイトの固定サイズレコードに丸める**
ことに限定される。下位層 (QUIC / TLS / UDP) の packetization・coalescing・
retransmission・ACK・path migration・リレー経由といった挙動により、外部観測者
から見える単位は「BlueWren のフレーム」とは一致しない。したがって本 ADR・
README・脅威モデルでは「ネットワーク上のパケットサイズを完全に隠す」とは
書かない。

## 決定

v0.2 では 256 バイト固定長フレーム (構造 v1) を採用する。

```
[version: 1B][type: 1B][len: 2B][payload: N bytes][padding: random]
└────────────────── 256 bytes total ──────────────────┘
```

フィールド:

- `version` (1 バイト): プロトコルバージョン (現在 `0x01`)。バージョン不一致は
  互換解釈を試みず、明示的なエラーで終了する。
- `type` (1 バイト): フレーム種別。
  - `0x01`: Message (通常メッセージ)
  - `0x02`: Dummy (keepalive、将来の cover traffic 予約)
  - `0x03`: Control (列挙コードのみ。本文は `payload[0]` の 1 バイト数値コード)
  - `0x04`: SasHandshake (SAS 用 nonce)
  - `0x80..=0xFF`: **予約範囲** (実験的・私的用途・派生プロジェクト向け)。
    標準実装はこの範囲のフレームを受信した場合に明示的に拒否する。
  - 上記以外の未知タイプは受信時に明示的に拒否する。
- `len` (2 バイト big-endian): payload の実際のバイト長
- `payload` (N バイト): フレーム内容
- `padding`: 残り (256 - 4 - N) バイトを暗号学的乱数で埋める

最大ペイロード: 256 - 4 = **252 バイト**

### Control フレームの型付け (設計指針 §6.5)

`Control` 型 (`0x03`) のペイロードは自由文字列にせず、列挙可能な小さな数値
コードとする。v1 で定義する Control コード:

- `0x01 = Close` (明示的なセッション終了通知)
- `0x02 = AckSas` (SAS 確認完了の応答。SAS-v2 の番号再入力 UX で双方の確認を
  表現するために使用。詳細は ADR-0004 v2)

将来コードを追加する場合は version bump (v2) と本 ADR の改訂を同時に行い、
互換解釈は試みない。自由文字列を Control に許すと、相手が任意のデータを
制御フレームとして送れてしまい後段のパースで脆弱性を生むため、コード列挙
方式を不変条件として維持する。

### 予約範囲の明示 (設計指針 §6.4)

将来の拡張のため、フレーム種別の `0x80..=0xFF` の範囲を実験的・私的用途の
ために予約する。標準実装はこの範囲のフレームを受信した場合に明示的に拒否
する。予約を文書化することで、フォークや派生プロジェクトが独自種別を安全に
追加できる経路を提供する。

### 選定理由

- 短文チャットでは 252 バイトでほぼ十分
- フレーム分割ロジックが不要 (1 メッセージ = 1 フレーム)
- type 情報により、本文・SAS・制御を明確に区別できる
- version フィールドにより、将来のプロトコル進化時に明示的な互換性エラーが出せる
- パディング部にランダムバイトを入れることで、暗号化スキームへの保険となる
- 予約範囲と Control コード列挙により、フレーム拡張時の互換性リスクを抑える

## 帰結

利点:

- アプリ層で各メッセージを 256 バイトの固定サイズレコードに丸めるため、
  復号後の peer 視点で本文の正確な長さは隠蔽される (中間観測者・relay には
  そもそも本文長は見えないが、レコード境界の単位で送信される)
- フレーム種別の追加・変更が version bump で安全にできる
- SAS / dummy / control の追加が可能になる
- 実装が単純で監査しやすい
- 予約範囲とコード列挙により、派生プロジェクトと標準実装の責務を分離できる

代償:

- 短いメッセージでも 256 バイトを送るため帯域効率は悪い
- 252 バイトを超えるメッセージは送信できない (アプリ層で拒否)
- 「ユーザーが何かを送信した」という事実 (タイミング) は隠せない
- 復号後の peer は本文の正確な長さを知る (relay や中間観測者は知らない)
- 下位層 (QUIC/TLS/UDP) のパケットサイズは隠蔽されない (設計指針 §6.2)

## 将来の検討事項

- 多段階パディング (256, 1024, 4096 バイトの三段階) で長文と短文の両立。
  ただし v0.2 非目標 (ADR-0001) のうち「ファイル送信」「タイミング解析耐性」と
  抵触しない範囲で扱う。
- cover traffic 導入後の固定間隔送信との組み合わせ (タイミング隠蔽)。
  これは ADR-0001 の非目標であり、導入時は ADR-0001 の更新を伴う。
- フレーム分割対応 (現状は 1 メッセージ = 1 フレームの 1:1 対応)。

これらは別 ADR を起こして検討する。

## 改訂履歴

- 2026-05-06 (v0.1): 初版 `[len: 2B][payload][padding]`
- 2026-05-06 (v0.2): version + type フィールドを追加した v1 構造に拡張
- 2026-05-06 (v0.2 audit-sync): 予約範囲 `0x80..=0xFF` の明示、Control コード
  (`0x01 Close`, `0x02 AckSas`) の列挙化、設計指針 §6.2 と整合した表現
  (パケットサイズ隠蔽とは表現しない) への修正。
