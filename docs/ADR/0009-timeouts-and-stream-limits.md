# ADR-0009: タイムアウトとストリーム数の構造的制限

- **ステータス**: 採択 (Accepted)
- **日付**: 2026-05-06
- **関連**: 設計指針 §5.3 §5.4 §5.6 / `docs/principles/THREAT_MODEL.md` A6
  (悪意ある peer) / ADR-0001 / ADR-0002 / ADR-0004

## コンテキスト

無期限に待機する非同期呼び出しは、サービス拒否 (DoS) 攻撃や WAN 不通時の
プロセス固着を招く。BlueWren は 1 対 1 短時間セッションを前提とするため、
すべての await 点に妥当な timeout を被せ、想定外の停滞をプロセス再起動で
解決できる状態に保つ必要がある。

また、Iroh の `Connection` は標準ではある程度のストリーム数を許容する。
1 対 1 通信に必要なのは 1 本の双方向ストリームのみであり、相手が追加の
ストリームを開いて資源を消費する経路は構造的に閉じるべきである。

QUIC の双方向ストリームは「開いた側が何かを書くまで相手側で `accept_bi()` が
完了しない」という仕様的特性を持つ。この順序を逆転させるとデッドロックが
起きる。

これらは設計指針 §5.3 / §5.4 / §5.6 で定義されているが、ADR としては
未記録だったため本 ADR で正式化する。

## 決定

### 全 await 点への timeout 必須化 (設計指針 §5.3)

以下の async 操作には例外なく timeout を被せる:

| 操作                                       | timeout 値の方針          | 備考                                     |
|--------------------------------------------|---------------------------|------------------------------------------|
| `Endpoint::builder(...).bind()`            | 10–15 s                   | バインド失敗時は即時エラー終了           |
| `endpoint.online()`                        | 10–15 s                   | リレー登録待機。失敗時は警告して終了     |
| `endpoint.accept()`                        | 対話的に長め (利用者操作) | 利用者が peer の到着を待つ用途           |
| `endpoint.connect(addr, ALPN)`             | 15–30 s                   | direct/relay 経由いずれも対象            |
| `conn.open_bi()` / `conn.accept_bi()`      | 15 s                      | 接続確立直後の単一ストリーム             |
| 各フレームの `read`                        | 30 s                      | `READ_FRAME_TIMEOUT`                     |
| SAS 確認入力 (利用者の番号再入力)          | 120 s                     | `SAS_CONFIRM_TIMEOUT`                    |
| idle (本文チャットでの無入力)              | 5 min (300 s)             | `IDLE_TIMEOUT`                           |
| session 全体の絶対上限                     | 1 h (3600 s)              | `SESSION_MAX_DURATION`                   |
| graceful close (`send.finish` / `stopped`) | 数秒程度の絶対上限        | プロセス終了がブロックされない値        |

これらは Rust 側の `tokio::time::timeout` を使い、超過時はセッションを終了
する (再試行は提供しない、ADR-0004 v2 と整合)。

### ストリーム数の構造的制限 (設計指針 §5.4)

`Connection` 確立直後に、以下の上限を設定する:

- **`max_concurrent_bi_streams = 1`**: 双方向ストリームは 1 本のみ
- **`max_concurrent_uni_streams = 0`**: 単方向ストリームは禁止

これにより、相手が追加ストリームを開いて資源を消費する経路が構造的に
閉じる。1 対 1 専用という設計上の制約を、トランスポート層の不変条件として
実装する。実 API シグネチャ (`TransportConfig` 経由か `Connection` 直接設定か)
は Iroh / Quinn のバージョンに依存するため、実装着手時に `docs.rs/iroh/latest`
と `cargo check` で確定する (ADR-0002 の規律に従う)。

### 接続シーケンスの順序保証 (設計指針 §5.6)

QUIC の双方向ストリームは、開いた側が何かを書くまで相手側で `accept_bi()` が
完了しない仕様である。したがって:

- **接続側 (`connect`)**: `open_bi()` 直後に最初のフレーム (SAS nonce) を
  送信する
- **待機側 (`listen`)**: `accept_bi()` を **timeout 付きで** 待つ

この順序を逆転させるとデッドロックが起きるため、両側の実装と統合テスト
(`tests/`) で本順序を不変条件として固定する。

### timeout 値の更新ポリシー

本 ADR で示した値は v0.2 時点の運用上の実用値であり、実測ベースで微調整
する余地はある。値を変更する PR は本 ADR を改訂し、変更理由 (DoS 抑制と
利用者体験のトレードオフのどちらに寄せるか) を明示する。

## 帰結

利点:

- どの非同期点でも無期限ブロックが発生せず、DoS 経路と WAN 不通時の固着を
  抑制できる
- ストリーム数 1/0 制限により、相手の ストリーム濫用による資源枯渇を構造的に
  防げる
- 接続シーケンス順序の固定により、デッドロックがコードベース上に再導入
  されない
- 設定値が ADR として一覧されるため、第三者監査時にチェック項目として参照
  しやすい

代償:

- timeout を passing する unit 引数が増え、コード署名が冗長になる
  (`Duration` 定数の集約で緩和)
- 1 対 1 専用という設計制約をトランスポート層に固定するため、将来の機能
  (グループチャット等) は ADR-0001 非目標を侵食しないかぎり追加できない
  (許容: 非目標は意図的な制約)
- Iroh / Quinn の API がバージョン更新で変わると本 ADR の実装手順も
  追随する必要がある

## 将来の検討事項

- timeout 値を CLI フラグ (`--idle-timeout` 等) で利用者がオーバーライド
  できる経路の妥当性 (DoS 耐性とトレードオフ)
- Iroh の TransportConfig 全体を `presets::N0` から派生させて BlueWren 専用
  preset 化する案
- 統合テストで「ストリーム濫用」「フレーム未送信での `accept_bi` 待機」を
  ファジングするケースの追加
