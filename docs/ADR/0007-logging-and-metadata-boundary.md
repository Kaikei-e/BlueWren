# ADR-0007: ログレベルとメタデータ境界

- **ステータス**: 採択 (Accepted)
- **日付**: 2026-05-06
- **関連**: 設計指針 §9 / `docs/principles/THREAT_MODEL.md` A7 (ローカル受動
  フォレンジック攻撃者) / ADR-0004 (チケット・SAS) / ADR-0005 (リレー)

## コンテキスト

`tracing` を含むログ出力は、運用とデバッグに不可欠な一方で、機微メタデータ
(EndpointId、ALPN、relay URL、チケット文字列、SAS、nonce) が混入すると、
プロセス終了後の scrollback / shell history / journald / 標準出力リダイレクト
ファイルに残り、ローカル受動フォレンジック攻撃者 (A7) に観測される経路に
なる。

`info` 以下のレベルはライブラリ側 (`iroh` / `quinn` / `tokio`) が機微情報を
出力する可能性があり、デフォルトでオンにすると意図しない漏洩を生む。

## 決定

### デフォルトログレベル (設計指針 §9.1)

通常運用のデフォルトログレベルは **`bluewren=warn,iroh=warn`** とする。
`info` レベルは `EndpointId` / 接続イベント / relay 経由の挙動など機微情報の
出力源となるため、デフォルトでオフにする。

`warn` 以上は本当に注意を要する事象 (タイムアウト、不正フレーム受信、
接続失敗、未知フレーム種別の拒否等) のみに使う。

### メタデータログのオプトイン (設計指針 §9.2)

EndpointId、リレー URL、相手アドレス、チケット文字列等のメタデータをログに
出力するには、**`--unsafe-debug-log-metadata`** フラグを明示的に指定する
必要がある。

- フラグ名に `unsafe-` プレフィクス (設計指針 §8.2) を付け、コマンドラインを
  見ただけで「これは安全な使い方ではない」と利用者に伝わるようにする。
- フラグ使用時は起動メッセージで明示的な警告を出力する。
- このフラグを使うとログにメタデータが残る。`tee` 等でファイルへ
  パイプすると永続化されるため、トラブルシューティング後にこれを残し続けない
  動機付けとなる。

### 機微フィールドのソース上での明示 (設計指針 §9.3)

ソースコード内で機微情報を扱う箇所には **`// [SENSITIVE]`** のコメントを
付ける。これにより:

- コードレビューや `grep -r '\[SENSITIVE\]' src/` で機微情報の取り扱い箇所が
  一覧できる。
- 新規メンバーが誤って `Debug` 出力やログマクロに渡してしまう事故を予防する。

「機微」と分類するフィールドの一覧 (本 ADR で固定):

- `EndpointId` (ローカル / リモートとも)
- `EndpointAddr` (relay URL / direct addresses 含む)
- ALPN 文字列 (ALPN 自体は機微ではないが、誤接続デバッグ時の混入を防ぐ)
- チケット文字列 (Iroh の `EndpointTicket` 表現)
- SAS 値 / SAS 計算入力 (nonce ペア、TLS exporter から導出した keying material)
- `bluewren=info` 以下のスパン名・属性のうち上記を含むもの

新たに機微フィールドを追加する PR は、本 ADR への追記と `// [SENSITIVE]`
の追加を必須とする。

### `Debug` 派生時の注意

機微フィールドを含む構造体に `#[derive(Debug)]` を付けない。`secrecy::SecretBox`
などのラッパは `Debug` 経由の意図しない露出を構造的に防ぐため、機微フィールドの
保持には可能な限りこれらを使う (ADR-0004 v2 の Zeroize 範囲と整合)。

## 帰結

利点:

- デフォルト動作で機微メタデータが scrollback / journald に残らない
- `--unsafe-debug-log-metadata` 命名で「危険なデバッグモード」が利用者に伝わる
- `// [SENSITIVE]` により機微フィールドが grep 可能になる
- 新規開発者が誤って機微情報を `Debug` / `tracing!` で出力する事故を予防する

代償:

- デバッグ困難な接続障害ではメタデータログを意図的にオンにする必要がある
- 機微の判定漏れがあると本 ADR の保護が成立しないため、レビュー段階で
  `// [SENSITIVE]` の付与を確認する文化が必要

## 将来の検討事項

- `tracing-subscriber` のフィルタを compile-time で固定し、機微フィールドを
  含むスパンを最初からビルドから外す経路 (より強い保証)
- ログ出力先を syslog / journald に分離して permissions を絞る案
- `ENV_FILTER` 経由でのオーバーライドを許可するか禁止するかの整理
