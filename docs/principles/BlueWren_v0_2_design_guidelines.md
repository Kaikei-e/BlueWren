# BlueWren v0.2 設計指針

> 二度のセキュリティ監査を経た現在の到達点を、コードや疑似コードに依存しない「設計原則」として記述する文書

**作成日**: 2026-05-06
**対象**: BlueWren v0.2.0 以降の全実装判断
**位置づけ**: 実装の選択肢が分岐したときの判断基準

---

## 文書の位置づけ

本文書は、コードを含まない設計指針 (design guidelines) として、BlueWren が将来どんな実装変更を経ても保つべき原則を記述する。具体的なコード例・関数シグネチャ・依存バージョン・モジュール構造は、本文書ではなく `src/` 配下のソースコードと、`docs/ADR/` 配下の ADR (Architecture Decision Record) で管理する。

本文書と他の文書との関係は次のとおりである。設計指針 (本書) は「何を守り、どう判断するか」という抽象原則を定める。脅威モデル (`THREAT_MODEL.md`) は「誰から、何を、どこまで守るか」という具体的な防御範囲を列挙する。ADR は「ある時点で、なぜ A ではなく B を選んだか」という判断記録を残す。安全な使い方ガイド (`SAFE_USAGE.md`) は「利用者が果たすべき周辺責務」を述べる。これら四種の文書は相互に参照されつつ、それぞれ独立した目的を持つ。

新機能の追加、依存関係の更新、API の変更、リファクタリングを行う際は、本文書のどの原則に該当するかを必ず確認する。原則と矛盾する変更を加える場合は、本文書を改訂したうえで実装を進める。実装が原則に追従するのではなく、原則が実装より先に確定している状態を維持する。

## 1. 中核となる設計哲学

### 1.1 誇大表現の徹底排除

「匿名性最優先」「監視に強い匿名通信」「フォレンジック痕跡を残さない」「中間者攻撃を防ぐ」「パケットサイズを隠蔽」といった包括的な強い主張は、BlueWren では使用しない。これらの表現は、技術的に完全達成することが極めて困難であるだけでなく、達成できると利用者に誤認させること自体が、結果として利用者を危険に晒す。たとえば「匿名通信」と信じて高リスク用途で使った利用者が、実は IP アドレスを露出させていた場合、誤った安心感は能動的な害をなす。

代わりに、提供している保護を具体的かつ限定的に列挙する。「短命 endpoint identity」「SAS 確認付き暗号化チャット」「アプリ層の固定長フレーム」「アプリ自身による履歴非保存」のように、何が保護されているかを利用者が事実として確認できる粒度で表現する。これは弱い表現ではなく、OSS として誠実で強い表現である。

### 1.2 脅威モデル先行主義

機能追加・変更・削除のすべての判断は、`THREAT_MODEL.md` との整合性検証から始める。コードを書く前に「この変更によって、どの攻撃者 (A0〜A7) への防御がどう変わるか」「新たに発生する残留リスクは何か」を文書として明示する。実装の都合で脅威モデルを後追いで書き換えることは原則として禁止する。脅威モデルが先、実装が後の順序を維持することで、機能追加のたびに防御範囲が暗黙のうちに後退する事態を防ぐ。

### 1.3 表面積の最小化

機能を増やすことは、レビュー対象を増やし、セキュリティ責務を分散させ、結果として安全性主張の検証コストを倍々で増加させる。BlueWren は学習・実験プロジェクトとして、利便性のための機能を意図的に却下する判断を維持する。グループチャット・ファイル送信・履歴保存・GUI・モバイル対応・複数識別子管理・連絡先発見はすべて非目標である。利用者がこれらを欲する場合、Briar・SimpleX・Signal などの別カテゴリのツールを推奨する。

### 1.4 公式ドキュメントを正とする原則

依存ライブラリ、特に `iroh` のような破壊的変更が多いクレートの API は、AI モデルの訓練データや過去のサンプルコードではなく、`docs.rs` の最新表示と `cargo check` の実コンパイル結果を正とする。BlueWren は QUIC・TLS・暗号プリミティブの上に構築されており、これらは半年で API が変わる領域である。記憶や雰囲気で実装した設計は、コンパイルすら通らないか、通っても期待した挙動を持たない。

## 2. セキュリティ表現の指針

### 2.1 禁止する表現

README、ランディングページ、機能説明、ブログ記事、ソーシャルメディアのいずれにおいても、以下の語句は使用しない。「完全匿名」「監視に強い匿名通信」「パケットサイズを隠蔽」「フォレンジック痕跡を残さない」「中間者攻撃を防ぐ」「安全な通信」「プライバシーを守る」。これらは抽象的かつ包括的であり、利用者がどこまで保護されているかを誤解させる。

### 2.2 推奨する表現

代わりに、事実ベースの限定的表現を使う。「短命 endpoint identity」「アプリ層の固定長フレーム」「SAS 確認付き相手認証」「アプリによるメッセージ・鍵の意図的非保存」「IP・通信タイミング・通信量は隠さない」「侵害された端末は保護しない」。これらは検証可能な事実であり、利用者が自分のリスクモデルに照らして適切性を判断できる。

### 2.3 残留リスクの併記原則

すべての防御主張に対し、対応する残留リスクを併記する。「QUIC TLS 1.3 で内容を秘匿する」と書くなら、「ただし通信タイミングと通信量は観測可能」を直近に書く。「SAS で MitM を検出可能」と書くなら、「ただし利用者が能動的に確認しなければ意味がない」を併記する。防御と残留リスクを切り離して記述すると、利用者は防御だけを読み、残留リスクを見落とす傾向がある。

## 3. アイデンティティ管理の指針

### 3.1 揮発性の維持

プロセス起動時に新規 Ed25519 鍵を生成し、終了時にメモリから消去する原則は v0.1 から維持する。ディスク・設定ファイル・OS キーリングへの保存は行わない。鍵リング・エクスポート・回転・失効といった鍵管理機能を一切持たないことで、鍵管理に伴う複雑性とそこから生じる脆弱性を構造的に排除する。代償として「同じ相手と再び話す」場合は毎回チケット交換と SAS 確認が必要になるが、これは BlueWren の用途である「短時間 1対1 セッション」と整合する。

### 3.2 長期鍵の取扱い

長期鍵の導入は v0.2 の範囲外である。将来検討する場合も、必ずオプトインとし、デフォルトは揮発のまま維持する。長期鍵を導入する場合は、保存先・暗号化方式・パスフレーズ要件・バックアップ・回転・失効の全側面を設計し、本文書と脅威モデルを大きく改訂する必要がある。これらを設計しないまま「便利だから」という理由で長期鍵を入れることは禁止する。

### 3.3 鍵を「使う」と「触れる」の境界

秘密鍵そのものは Iroh のエンドポイント API 内部で管理され、アプリケーションコードが鍵バイト列を直接扱う場面は最小化する。鍵を直接扱うほど、メモリ上の残留・ログへの混入・デバッガでの露出といった経路で漏洩するリスクが高まる。アプリ側で鍵を扱う必要が生じた場合は、その必要性を ADR で正当化したうえで、`Zeroizing` や `secrecy::SecretBox` のラッパで保護する。

## 4. SAS / ペアリング確認の指針

### 4.1 SAS は安全性主張の中核として扱う

チケット交換経路の侵害を利用者が能動的に検出する唯一の手段が SAS (Short Authentication String / Pairing Code) である。したがって、SAS の設計は BlueWren の安全性主張全体の強度を決定する。SAS が弱ければ、たとえ他のすべての要素が完璧でも、MitM 検出の保証は崩れる。SAS の設計判断は他のどの設計判断よりも保守的に行う。

### 4.2 ビット長の下限と推奨

SAS の有効ビット長は最低 24 ビット、推奨 30 ビットとする。20 ビット (約 100 万通り) は単発の目視確認として機能するが、安全性主張の中核としては弱い。30 ビット (約 10 億通り) であれば、攻撃者が SAS を偶然一致させる確率は 10^-9 のオーダーになり、現実的な防御として通用する。UX が重い場合の譲歩線は 24 ビットとし、それ以下には下げない。

### 4.3 TLS セッションエクスポーターへのバインド

SAS の根として、QUIC セッションごとに一意な秘密値を引き出す TLS exporter (RFC 5705 / RFC 8446) を使用する。Iroh の `Connection` には `export_keying_material` メソッドがあり、双方が同じ label と context を指定すると、暗号学的に強い同一の擬似乱数列を独立に得られる。これにより SAS が単なる EndpointId と nonce のハッシュから「実際の TLS セッションそのものに紐付いた値」へと格上げされる。Iroh は内部に Ed25519 鍵による相互認証を持つため、TLS exporter ベースの SAS は MitM 攻撃者が同じ値を導出することを困難にする。

### 4.4 確認 UX は番号再入力方式

利用者が漫然と Enter キーを押して確認をスキップする事故を構造的に防ぐため、表示された SAS をそのまま入力させる方式を採用する。ユーザーは画面に表示された値を、別経路 (通話・対面など) で相手の値と照合し、一致した場合に同じ値をプロンプトに打ち込む。一致しない値が入力された場合はセッションを直ちに終了する。Enter のみで通す UX は誤操作・確認漏れに対して脆弱であり、SAS の意義そのものを毀損するため採用しない。

### 4.5 不一致時のアボート経路

SAS が不一致と判明した場合、利用者にはセッション中断と接続解除を即座に促し、自動的な再試行は一切行わない。再試行は攻撃者にとって SAS の偶然一致を試行回数で稼ぐ機会を提供しうるため、明示的に禁止する。利用者にはチケット交換経路自体を見直すよう案内する。

### 4.6 SAS 確認前の本文送信禁止

SAS 確認が完了するまで、`Message` 型のフレームは一切送信しない。これは構造的な不変条件として実装に組み込む。確認前にメッセージを送ると、たとえ短い挨拶であっても攻撃者が読んでしまう可能性がある。SAS 確認は「相手が想定どおりかを確認する手続き」であり、確認が終わるまでは「相手が想定どおりとは限らない」前提で動作する。

## 5. Iroh トランスポート統合の指針

### 5.1 公式 API への厳密な追随

`iroh` クレートの API は破壊的変更が多く、訓練データや過去のサンプルコードに頼った実装は通用しない。実装着手前に必ず `docs.rs/iroh/latest` を確認し、`cargo check` でコンパイル可能性を検証する。`NodeId`/`NodeAddr` といった旧名称は廃止され、現在は `EndpointId`/`EndpointAddr` である。`endpoint.node_addr()` は廃止され、`endpoint.addr()` または `endpoint.watch_addr().get()` が現行である。`conn.local_endpoint_id()` のようなメソッドは存在せず、ローカル ID は `Endpoint::id()` で、リモート ID は `Connection::remote_id()` で取得する。これらは公式 docs の確認と `cargo check` でしか検証できない。

### 5.2 Builder + ALPN 先行設定

エンドポイントの構築は短縮形 (`Endpoint::bind().await`) ではなく、Builder パターン経由で行う。`Endpoint::builder(presets::N0)` のように preset を指定し、`alpns` を bind 前に設定する。ALPN を `bind` 後に追加する経路は、レース条件を生む可能性があり、また「どの ALPN を受け入れるか」が構築時の不変属性として明示されるべきだからである。

### 5.3 すべての非同期 I/O に timeout を被せる

無期限に待機する非同期呼び出しは、サービス拒否攻撃や WAN 不通時のプロセス固着を招く。BlueWren ではすべての await 点に妥当な timeout を被せることを必須とする。具体的には `endpoint.online()`、`endpoint.accept()`、`endpoint.connect()`、`conn.open_bi()`、`conn.accept_bi()`、各フレームの `read`、SAS 確認入力、graceful close 待機、これら全てに timeout を設定する。timeout 値は用途ごとに異なるが、起動系は短く (10〜15 秒)、対話系は長く (5〜10 分)、セッション全体には絶対上限 (1 時間) を設ける。

### 5.4 ストリーム数の構造的制限

1対1 通信に必要なのは 1 本の双方向ストリームのみである。Iroh の `Connection` には同時ストリーム数の上限を変更する API があり、BlueWren では接続確立直後に `max_concurrent_bi_streams` を 1 に、`max_concurrent_uni_streams` を 0 に設定する。これにより、相手が追加ストリームを開いて資源を消費する経路が構造的に閉じられる。1対1 専用という設計上の制約を、トランスポート層の不変条件として実装する。

### 5.5 リレー観測の脅威モデル化

デフォルトリレー (Number Zero, Inc. 運用の n0 preset) を使用する場合、リレー運用者は EndpointId ペア・タイムスタンプ・おおよそのトラフィック量を観測できる。これは技術的な必然であり隠蔽不可能であるため、`THREAT_MODEL.md` の A3 に明記する。「Iroh モードは IP 匿名化を提供しない」を README にも明示する。将来 Tor / I2P モードを追加する場合は、Iroh モードと混合せず、別サブコマンド (`bluewren onion listen` 等) として分離する。同じ「匿名性」というラベルで複数のモードをまとめると、それぞれの脅威モデルが曖昧になる。

### 5.6 接続シーケンスの順序保証

QUIC の双方向ストリームは、開いた側が何かを書くまで相手側で `accept_bi()` が完了しない仕様である。したがって接続側は `open_bi()` 直後に最初のフレーム (SAS nonce) を送信し、待機側は `accept_bi()` を timeout 付きで待つ。この順序を逆転させるとデッドロックが起きる。

## 6. フレームプロトコルの指針

### 6.1 アプリ層メッセージ単位の固定長

アプリケーションが書き出すメッセージ単位を固定長 (256 バイト) に揃える原則は維持する。これにより、復号後の peer 視点では各メッセージの本文長が常に同じになり、長文・短文の区別が peer 側にも残らない。短いメッセージでも 256 バイトを送るため帯域効率は悪いが、テキストチャットの規模では問題にならない。

### 6.2 「パケットサイズ隠蔽」と表現しない

アプリ層で 256 バイトのフレームに揃えても、QUIC・TLS・UDP の packetization、coalescing、retransmission、ACK、path migration、リレー経由といった下位層の挙動により、外部観測者から見える単位は「BlueWren のフレーム」とは一致しない。したがって README や脅威モデルでは「アプリ層メッセージを一定サイズのレコードに丸める」と表現し、「ネットワーク上のパケットサイズを完全に隠す」とは書かない。実際に守れている範囲を超えた表現を慎む。

### 6.3 バージョンとタイプによる構造化

フレーム先頭の最初のバイトをプロトコルバージョン、次のバイトをフレーム種別とする。バージョンが一致しない場合は明示的なエラーで終了し、互換解釈を試みない。種別は Message・Dummy・Control・SasHandshake の四種を持つ。型を持たせることで、SAS 用の nonce 交換と本文メッセージが同一ストリームを安全に共有でき、将来の cover traffic や制御信号の追加にも構造的に対応できる。

### 6.4 予約範囲の明示

将来の拡張のため、フレーム種別の `0x80..=0xFF` の範囲を実験的・私的用途のために予約する。標準実装ではこの範囲のフレームを受信した場合に明示的に拒否する。予約を文書化することで、フォークや派生プロジェクトが安全に独自種別を追加できる。

### 6.5 制御フレームの型付け

`Control` 型のペイロードは自由文字列にせず、列挙可能な小さな数値コードとする。たとえば `0x01 = Close`、`0x02 = AckSas` のように、有限の意味を持つコードとして定義する。自由文字列を許すと、相手が任意のデータを制御フレームとして送れてしまい、後段のパースで脆弱性を生む可能性がある。

## 7. 端末出力の指針

### 7.1 改行のみ許可、CR は禁止

受信メッセージを表示する前のサニタイズで許可する制御文字は、`\n` (LF) と `\t` (HT) のみとする。`\r` (CR) は行頭復帰を起こし、画面上の偽装 (UI spoofing) に使えるため、削除または視覚化 (`\r` という文字列に置換) する。`\n` も複数行を使った偽プロンプト表示に使えるため、後述のとおり各行プレフィクスで防御する。それ以外の C0/C1 制御文字はすべて U+FFFD (置換文字) に変換する。

### 7.2 マルチライン受信表示の各行プレフィクス

受信メッセージが複数行にまたがる場合、最初の行だけでなく 2 行目以降にも `[相手]` のプレフィクスを付ける。これにより、相手が「`改行 > sudo rm -rf ...`」のようにシェルプロンプトを偽装する経路を遮断する。プレフィクスは利用者が「これは相手から来たデータである」と視覚的に区別できる手がかりとして機能する。

### 7.3 エスケープシーケンスの全消去

ESC (`0x1B`) で始まるシーケンスは、CSI (`ESC [ ...`)・OSC (`ESC ] ...`)・その他の単発エスケープすべてを消費して破棄する。CSI は画面クリアやカーソル制御に、OSC はクリップボード書き換え (OSC 52) や端末タイトル変更 (OSC 0/2) に悪用される。完璧なパーサは不要だが、既知の悪用パターンに対する実用的な防御を提供する。

### 7.4 自前パーサと外部クレートの選択

`strip-ansi-escapes` のような外部クレートを使う選択肢もあるが、BlueWren の規模 (数十行のパーサ) では監査性を優先して自前実装を採用する。外部クレートは依存関係を増やし、その crate 自体の脆弱性に左右される。50 行程度の単純なパーサであればコードレビューで挙動を完全に把握でき、テストで主要な悪用パターンに対する防御を直接検証できる。

## 8. CLI / UX の指針

### 8.1 高感度情報をコマンドラインに載せない

チケットを `argv` で渡す経路は、shell history、`/proc/<pid>/cmdline`、process list、screen/tmux ログ、各種コマンドロガー、クラッシュレポートへの漏洩を生む。BlueWren のチケットは secret ではないが、漏洩すると意図しない第三者が接続を試せる高感度情報であり、引数渡しは構造的欠陥である。デフォルトは stdin プロンプトとし、パイプ入力は `--ticket-stdin` フラグで明示的に指定する。

### 8.2 危険な選択肢は明示的なプレフィクス

互換のために残す危険な選択肢は、`--unsafe-` プレフィクスを付ける。たとえば引数経由のチケット入力は `--unsafe-ticket-arg`、メタデータをログに出すモードは `--unsafe-debug-log-metadata` とする。プレフィクスにより、コマンドラインを見ただけで「これは安全な使い方ではない」と利用者に伝わる。さらに該当フラグの使用時は起動メッセージで明示的な警告を出力する。

### 8.3 起動時の安全注意の表示

毎回の起動時に短い安全注意を表示し、利用者が「OS 側の痕跡は別問題」であることを継続的に意識する。表示内容は冗長にせず、「BlueWren はメッセージ・鍵をディスクに保存しないが、OS・ターミナル・clipboard・swap 等は別問題である。詳細は SAFE_USAGE.md を参照」程度に抑える。長すぎる注意は無視されるため、要点を短く繰り返し表示する設計とする。

## 9. ログとメタデータ取扱いの指針

### 9.1 デフォルトは warn 以下

通常運用のデフォルトログレベルは `bluewren=warn,iroh=warn` とする。`info` レベルですら EndpointId 等の機微情報が出る可能性があり、デフォルトでオンにすべきではない。warn 以上は本当に注意を要する事象 (タイムアウト・不正フレーム受信・接続失敗) のみに使う。

### 9.2 メタデータログのオプトイン

EndpointId、リレー URL、相手アドレス、チケット文字列等のメタデータは、`--unsafe-debug-log-metadata` フラグを明示的に指定した場合のみログに出力する。このフラグを使うとログにメタデータが残り、shell コマンドのパイプで `tee` などに繋ぐと永続化される。フラグ名にも `unsafe` を含めることで、利用者がトラブルシューティング後にこれを残し続けない動機付けとする。

### 9.3 機微フィールドのソース上での明示

ソースコード内で機微情報を扱う箇所には `// [SENSITIVE]` のようなコメントを付ける。これによりコードレビューや grep で機微情報の取り扱い箇所が一覧でき、新規メンバーが誤って `Debug` 出力やログマクロに渡してしまう事故を予防する。

## 10. メモリ衛生の指針

### 10.1 zeroize の適用範囲

機微値は `zeroize::Zeroizing` または `secrecy::SecretBox` のラッパで包む。具体的には、入力されたチケット文字列、SAS 用の nonce、SAS 値そのもの、エクスポートされた keying material が対象である。これらは Drop 時にメモリがゼロ埋めされる。受信バッファや stdin バッファは本質的にすべての入力を経由するため、完全な保護は難しいが、可能な範囲で best-effort のゼロ化を試みる。

### 10.2 Rust の所有権モデルとの妥協

Rust の所有権・move セマンティクス・allocator の挙動・コンパイラ最適化により、メモリのゼロ化は本質的に best-effort である。`zeroize` クレートは「最適化で消されにくい形で書き込みを行う」プリミティブを提供するが、move されたコピー、allocator が再利用したページ、OS のスワップに書き出されたデータまでは制御できない。`mlock` や `mprotect` のような OS レベルの保護を導入する選択肢もあるが、その複雑性は v0.2 の範囲を超える。

### 10.3 限界の明示

メモリフォレンジック耐性 (RAM ダンプからの鍵・メッセージ復元への対策) は提供しないことを `SECURITY.md` と `THREAT_MODEL.md` に明記する。「best-effort で zeroize する」という主張は、「メモリフォレンジックを防ぐ」という主張ではない。両者の違いを文書で明確にし、利用者が誤解しない表現を選ぶ。

## 11. 依存関係とサプライチェーンの指針

### 11.1 Cargo.lock の必須コミット

BlueWren は CLI アプリケーションであるため、`Cargo.lock` は必ずリポジトリにコミットする。ライブラリと異なり、配布されるバイナリは特定の依存解決結果に基づいてビルドされるべきである。lockfile がない状態で「監査済み」とは決して言えない。lockfile は依存関係の再現性とサプライチェーン監査の前提である。

### 11.2 cargo-audit と cargo-deny の CI 必須化

毎日定時に `cargo audit` と `cargo deny check` を CI で実行する。これにより、新しい RustSec advisory が公開された日のうちに影響を検知できる。`cargo deny` は advisory に加え、ライセンス整合性、依存ソースの正当性、wildcard 依存の禁止などをチェックする。advisory を `deny`、yanked を `deny`、unmaintained を `warn` に設定する。これらが PR で fail する場合は、修正なしにマージしない。

### 11.3 quinn-proto の transitive 監視

QUIC ベースである以上、`quinn-proto` のバージョンには特に注意する。RUSTSEC-2026-0037 (`>=0.11.14` で修正済み) のような panic-on-untrusted-input 脆弱性は、BlueWren のような P2P チャットアプリでは深刻な DoS 経路となる。`cargo tree -i quinn-proto` を CI に組み込み、解決された具体的なバージョンを毎ビルドで記録する。

### 11.4 cargo-auditable による配布バイナリ監査可能性

リリースバイナリは `cargo auditable build --release --locked` でビルドする。これによりバイナリそのものに依存関係情報が埋め込まれ、後から `cargo audit bin` で配布バイナリの脆弱性チェックが可能になる。これは利用者がバイナリを受け取った時点でも「このバイナリの依存関係は監査可能である」という保証を提供する。

## 12. ドキュメンテーションの指針

### 12.1 コードと THREAT_MODEL の同期義務

機能追加 PR は必ず `THREAT_MODEL.md` の更新を含む。脅威モデルに影響しない PR は「実装の改善」として認められるが、影響する場合は脅威モデル更新なしでマージしない。これは「機能を入れたら、それが守れているもの・守れていないものを文書化する」という規律を維持するためである。

### 12.2 ADR による設計判断の追跡

設計上の重要な分岐 (採用した方式と却下した方式) は ADR として記録する。採用された案だけでなく、却下された案とその理由も書く。これにより、半年後に同じ判断を再検討する際、過去の議論を再生できる。ADR は採択・却下・置換 (superseded) のステータスを持ち、後続の ADR が前の ADR を上書きする場合は明示的にリンクする。

### 12.3 SAFE_USAGE.md による利用者への期待値設定

利用者に求める運用上の hardening 手順は `SAFE_USAGE.md` に集約する。shell history の一時無効化、core dump の無効化、swap・hibernation・clipboard・IME 履歴への注意、利用後のクリーンアップチェックリストといった、BlueWren アプリでは制御不能だが利用者環境で対処可能な事項を網羅する。これは「BlueWren が守れない部分を、利用者が補う」役割分担を明確にする文書である。

## 14. 実装フェーズの優先順位

### Phase 0: 実装現実性の検証

最初に行うべきは、設計ではなく実装可能性の確認である。ここを飛ばすと、設計だけが進んで実装段階で API 不一致やバージョン非互換に直面し、設計をやり直す手戻りが発生する。具体的には、`iroh` クレートを最新版に更新し、`cargo generate-lockfile`、`cargo check`、`cargo tree -i quinn-proto` を実行して、想定する API がコンパイル可能であることを確認する。このフェーズの完了をもって、以降の Phase の前提が固まる。

### Phase 1: 公開表現の修正

文書側の誇大表現を是正する。README、`THREAT_MODEL.md`、`SAFE_USAGE.md`、`SECURITY.md` を整備し、`cargo audit` / `cargo deny` の CI ワークフローを導入する。このフェーズはコード変更を伴わないため、Phase 0 と並行して実施可能である。

### Phase 2: セッション挙動の正しさ

Iroh エンドポイントの構築を Builder + ALPN 先行設定に統一し、`online()` を含むすべての非同期 I/O に timeout を被せる。チケット入力を stdin デフォルトに変更し、デフォルトログレベルを warn に下げる。ストリーム数を 1 双方向 / 0 単方向に制限する。

### Phase 3: SAS v2

nonce 交換、TLS exporter ベースの SAS 計算、30 ビット (または 24 ビット) のフォーマット表示、コード再入力による確認、不一致時のアボート経路を実装する。手動の二プロセステストで、同一 EndpointId では SAS が一致し、異なる EndpointId では一致しないことを検証する。

### Phase 4: 端末・ローカル衛生

サニタイズに CR の禁止と複数行プレフィクスを追加する。チケットバッファの `Zeroizing` 化、stdin と受信バッファの best-effort ゼロ化を実装する。

### Phase 5: フレーム v1

フレームプロトコルにバージョン・タイプフィールドを導入する。タイプの予約範囲を明文化し、Control コードを列挙型として定義する。バージョン不一致と未知タイプの拒否テストを追加する。

## 15. 最終的な安全性主張

BlueWren v0.2.0 の安全性主張は、本文書のすべての原則を踏まえて、以下の一文に集約する。

> BlueWren v0.2.0 は、短命 endpoint identity、帯域外 ticket、SAS 確認、非永続化方針、固定長アプリケーションフレームを組み合わせ、短時間の 1対1 暗号化チャットにおける内容秘匿と誤接続検出を補助する。ただし、IP・通信タイミング・通信量・relay metadata・端末侵害・OS 側痕跡は保護しない。

この表現は弱い表現ではなく、OSS として誠実で強い表現である。利用者は自分のリスクモデルに照らして BlueWren が適切かを判断でき、適切でない場合は別カテゴリのツール (Briar、SimpleX、Tails OS など) を選べる。誠実な制限の宣言が、利用者の自律的判断を支える。
