# BlueWren 脅威モデル

本文書は、BlueWren v0.2 が「何から何を守るのか」と「何を守れないのか」を
明示的に記述する。コード変更の際は、必ず本文書との整合性を検証すること。

## 想定する利用シナリオ

二人の利用者が、それぞれ別の物理マシン上で BlueWren を起動し、帯域外の
信頼できる経路（直接対面、別の暗号化済みチャネルなど）でチケットを交換し、
短時間（数分〜数時間、最大1時間で自動終了）の対話を行うシナリオを想定する。

## 防御対象 (assets)

| 防御対象              | 概要                                                                    |
| --------------------- | ----------------------------------------------------------------------- |
| メッセージ内容        | 利用者が入力する平文テキスト                                            |
| メッセージ長          | 各メッセージのバイト数                                                  |
| セッション識別        | 過去のセッションと現在のセッションの暗号学的紐付け                      |
| アプリ層フォレンジック痕跡 | アプリ自身がディスクに残す情報                                       |
| チケット              | 接続先 endpoint/address/relay 情報を含む接続能力                        |
| ペアリング確認情報    | SAS / fingerprint の一致性                                              |

## 想定する攻撃者 (adversaries)

### A0: チケットを入手した第三者

チケットを盗み見たり、帯域外経路上で差し替えたりできる攻撃者。

**防御**:
- ticket をコマンドライン引数に載せない（stdin プロンプト経由）。互換用 flag は
  `--unsafe-ticket-arg` のみで、起動時に明示警告を出す (ADR-0004 v2)
- 接続後に SAS (`XXXXX-XXXXX` 形式の **30 bit 値**、TLS exporter (RFC 5705) ベース) を
  表示し、利用者が別経路で照合する (ADR-0004 v2、設計指針 §4.2 / §4.3)
- 確認 UX は **番号再入力方式**: 表示された SAS をそのままプロンプトに入力させる。
  Enter のみで通す UX は採用しない (設計指針 §4.4)
- SAS 確認前は本文メッセージを送受しない (構造的不変条件、設計指針 §4.6)
- SAS 不一致 / 確認 timeout / Ctrl+C / EOF はセッションを即終了。自動再試行は
  提供しない (設計指針 §4.5)

**残留リスク**: 利用者が SAS を別経路で照合せず、画面表示の値をそのまま再入力
した場合、攻撃者との会話を検出できない (再入力 UX は確認スキップ事故を構造的に
減らすが、利用者の能動的照合がなければ意味がない)。

### A1: パッシブネットワーク観測者

インターネット中継網の任意の地点でパケットを観測できる。
内容を改竄したり接続を遮断したりはしない。

**防御**:
- QUIC/TLS 1.3 によるエンドツーエンド暗号化
- 256B 固定長アプリケーションフレームによる単一メッセージ本文長の隠蔽

**残留リスク**:
- 通信タイミング、フレーム数、総通信量、QUIC レベルのオーバーヘッド
- IP アドレス
- リレー利用の事実

### A2: アクティブネットワーク攻撃者

パケットの改竄、注入、遮断を行える。中間者攻撃 (MitM) を試みる。

**防御**:
- QUIC ハンドシェイクでの両者 Ed25519 公開鍵 (EndpointId) による相互認証
- ALPN 一致チェック (`/bluewren/0.2`、ADR-0002 の Builder + ALPN 先行設定で固定)
- A0 と同様の SAS 確認 (TLS exporter ベース 30 bit、番号再入力)。攻撃者が双方と
  独立に TLS を張った状態で同一の SAS を導出することは TLS exporter バインドに
  より困難となる (ADR-0004 v2)

**残留リスク**: チケット交換経路自体が侵害されている場合、A0 の残留リスクに帰着。

### A3: 中継リレー観測者

Iroh のデフォルトリレー (Number Zero 運用) を運用する者、もしくはそれを侵害した者。

**防御**: リレーは暗号化されたパケットしか見ない。本文は秘匿される。

**残留リスク**:
- どの EndpointId とどの EndpointId が通信しているか
- いつ通信したか、おおよそのトラフィック量
- relay fallback が継続しているか
- セッション中の二者の同時性

EndpointId は揮発的であるため過去のセッションとの紐付けは困難だが、
セッション中の同時性は明らかになる。独自リレー運用 (`--relay-mode custom`) や
Tor 統合は v0.3 以降で検討する (ADR-0005 参照)。

### A4: 物理マシンの侵害者

利用者のマシンへのリモートまたは物理アクセスを得た攻撃者。

**防御なし**。エンドポイントが侵害された時点で BlueWren の保護はすべて
無効化される。これは設計の前提として明示する。

### A5: 司法的開示命令

リレー運用者、ISP、利用者本人に対する法的な情報開示要求。

**部分的防御**:
- アプリはディスクに何も書かないため、押収後のディスク解析でメッセージは復元できない
- ターミナル / OS / clipboard / swap / coredump への痕跡は対象外 (A7 参照)

**残留リスク**:
- メモリフォレンジック (RAM ダンプ) からの鍵・メッセージ復元
- リレー運用者は接続メタデータ (EndpointId ペア、タイムスタンプ) を保持している可能性

### A6: 悪意ある peer

接続相手そのものが悪意を持つ。terminal escape sequence、過大入力、
接続維持、社会工学的誘導を行う。

**防御**:
- 受信メッセージ表示前に制御文字 / ANSI escape sequence を sanitize (ADR-0006)。
  許可制御文字は `\n`, `\t` のみ。`\r` は禁止。受信複数行は各行 `[相手]`
  プレフィクス付きで表示
- frame validation (version / type / length 検証、予約範囲 `0x80..=0xFF` を拒否、
  ADR-0003)
- 全 await 点に timeout: read 30s / idle 5min / session max 1h、SAS 確認入力 120s、
  endpoint online 10〜15s、connect / open_bi / accept_bi 15〜30s (ADR-0009)
- ストリーム数の構造的制限: `max_concurrent_bi_streams = 1`,
  `max_concurrent_uni_streams = 0` (ADR-0009)
- SAS 確認前は本文メッセージを処理しない (設計指針 §4.6)

**残留リスク**: 利用者が相手に送った平文は相手側で保存・転送・スクリーンショット可能。

### A7: ローカル受動フォレンジック攻撃者

プロセス終了後に端末、shell history、scrollback、swap、coredump、logs、
clipboard manager を調べる攻撃者。

**防御**:
- アプリ自身は履歴を保存しない (ADR-0001 非目標)
- ticket は stdin 経由で受け取り、`Zeroizing<String>` に格納 (ADR-0004 v2)
- nonce / SAS 値 / TLS exporter から導出した keying material / stdin バッファを
  `Zeroizing` または `secrecy::SecretBox` で包む (ADR-0004 v2、設計指針 §10.1)
- デフォルトログレベルは `bluewren=warn,iroh=warn` (ADR-0007)。EndpointId・
  relay URL・チケット等のメタデータは `--unsafe-debug-log-metadata` 必須。
  機微フィールドはソース上 `// [SENSITIVE]` 注釈で grep 可能化

**残留リスク**: OS / terminal / clipboard / swap / hibernation / IME 履歴は
BlueWren 単体では完全に制御できない。Rust の所有権・allocator 挙動・コンパイラ
最適化により zeroize は本質的に best-effort であり、メモリフォレンジック耐性は
提供しない (設計指針 §10.2 / §10.3)。利用者向けの hardening 手順は
[SAFE_USAGE.md](SAFE_USAGE.md) を参照。

## 明示的な非目標 (non-goals)

以下は v0.2 では達成しない。

- ネットワーク匿名性 (IP 秘匿)
- 交通解析耐性 (timing / volume / session duration の隠蔽)
- グローバルパッシブ攻撃者への耐性
- relay operator からのメタデータ秘匿
- 端末全体のフォレンジック耐性
- 相手 peer によるメッセージ保存・転送防止
- 完全なメモリフォレンジック耐性
- 多対多通信
- ファイル送信
- メッセージ履歴

## 変更管理

新機能を追加する際は、以下を義務付ける。

1. 本脅威モデル文書のどの項目に影響するかを評価する
2. 新たに発生する残留リスクを記述する
3. 必要であれば ADR を起票し、設計判断を残す
4. ADR / 設計指針 (`BlueWren_v0_2_design_guidelines.md`) / `SECURITY.md` /
   `SAFE_USAGE.md` / `README.md` の整合性を PR で確認する。
   設計指針 §13 (v0.2.0 リリースゲート) のチェック項目との対応を明示する。
